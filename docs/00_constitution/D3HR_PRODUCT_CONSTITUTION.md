# D3HR Product Constitution

## Immutable Identity

D3HR is a Public HR Evidence OS for Korean public institutions.

It connects daily work, performance evidence, competency data, NCS-based job data, career development, education, placement, evaluation, and public-sector management-evaluation evidence.

## One Sentence That Must Not Drift

D3HR turns public-institution employees' daily work, performance, and competency data into job-centered evidence for evaluation, placement, education, career development, and management-evaluation reporting.

## Founding Beliefs

- Job is the operating language of public HR.
- Performance must be evidenced, not remembered.
- Evaluation must be explainable and auditable.
- Career development must be designed, not left to accidental rotation.
- Low performance must be decomposed into job fit, competency, workload, goal clarity, management, and structure.
- High performance must become an institutional asset.
- AI may recommend, but humans and policy decide.
- Public HR must survive regulation, procedure, audit, and external evaluation.

## Product Boundaries

D3HR is not an ERP replacement first. ERP is the personnel record ledger. D3HR is the job, performance, and career decision engine above it.

D3HR is not a generic HR SaaS. It is specialized for Korean public institutions, NCS, regulation-heavy HR, retirement-security workforces, rotational placement, management evaluation, and audit evidence.

## Non-Negotiables

- Job-centered data model
- Evidence-based decisions
- Tenant isolation
- RBAC + ABAC + workflow-based access control
- Audit logs for sensitive reads, writes, approvals, and exports
- Versioned job descriptions
- Structured job-description element catalog
- Strict separation of product schema, generic dummy data, institution-specific inputs, and reference samples
- Explainable recommendation evidence
- Documentation synchronized with implementation

## Prohibited Patterns

- AI-only final evaluation decisions
- Automatic low-performer labeling
- Unlimited tenant-specific code branches
- Undocumented API or table creation
- Community-first product drift
- Dashboard-only implementation without evidence data
- Treating a real institution's sample input as D3HR default product content
